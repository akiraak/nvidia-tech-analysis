なぜ大量の GPU が必要なのか？

大規模言語モデル（LLM）のトレーニングには膨大なメモリと計算能力が必要で、1 台の GPU では到底足りない。

メモリが足りない

トレーニング時は、モデルの重み・勾配・オプティマイザの状態を全て GPU メモリに保持する必要がある。1 パラメータあたり約 16 バイトが必要（重み 2B + 勾配 2B + Adam optimizer 12B）。

キーコンセプト解説

このページで頻出する技術用語を、例え話を交えて解説する。

RDMA

Remote Direct Memory Access

あるコンピュータのメモリから別のコンピュータのメモリへ、CPU を介さず直接データを転送する技術。

通常の TCP/IP ではデータが何回もコピーされ CPU が処理に忙殺されるが、RDMA ではゼロコピーで直通転送。レイテンシは TCP/IP の数十〜数百 µs に対し、RDMA は 1〜5 µs。

従来の TCP/IP が「宅配業者が何度も中継する配送」なら、RDMA は「送り主から届け先まで直通のベルトコンベア」。CPU（宅配業者）は他の仕事に専念できる。

GPUDirect RDMA

GPU メモリへの直接アクセス

通常の RDMA でも GPU → CPU メモリ → NIC の経路が必要だった。GPUDirect RDMA は NIC が GPU メモリ（VRAM）に直接読み書きし、CPU メモリを一切経由しない。

実績: データ処理性能 74% 向上、CPU 使用率 43% 削減。

ベルトコンベアが倉庫（CPU メモリ）を経由せず、工場の生産ライン（GPU）に直接つながるイメージ。

Collective Communication

集団通信（AllReduce, AllGather 等）

複数の GPU が協調してデータを交換・集約する通信パターン。AI トレーニングでは全 GPU の勾配（学習結果）を合計して平均化する AllReduce が最も頻繁に使われる。

Llama 3.1 405B では毎イテレーション 約 810 GB の勾配データを全 GPU 間で交換する必要がある。

30 人のクラスで全員の平均点を計算する場面。AllReduce = 全員の点数を集めて合計し、結果を全員に配る操作。全員の提出を待つ必要があるため、最も遅い 1 人が全体のボトルネックになる。

NVLink vs PCIe

GPU 間の高速接続 vs 汎用バス

PCIe は CPU と周辺機器をつなぐ汎用規格。NVLink は GPU 間専用の高速インターコネクト。NVLink 5 は PCIe Gen5 の 14 倍の帯域幅（1.8 TB/s vs 128 GB/s）。

100 GB の転送: PCIe = 0.78 秒、NVLink = 0.056 秒。毎イテレーション数百回の交換ではこの差が致命的。

PCIe が「一般家庭の水道管」なら、NVLink は「工業用の大口径パイプ」。AI トレーニングは「プール何杯分もの水を秒単位で移動させる」必要があり、家庭用水道管では追いつかない。

SHARP

In-Network Computing

InfiniBand スイッチ上で AllReduce 等の計算を直接実行する技術。各 GPU がデータを 1 回だけ送信し、スイッチが途中で集約するため、ネットワーク上のデータ量が約半分に削減。性能は 2〜5 倍向上。

全国投票の集計で、全票を中央に送る代わりに各地方選管がその場で集計し、集計結果だけを上位に送る方式。中央に届くのは票そのものではなく集計数だけなので、郵便物の量が大幅に減る。

RoCE

RDMA over Converged Ethernet

標準 Ethernet 上で RDMA を実現するプロトコル。InfiniBand の専用ネットワーク不要で既存インフラを活用可能。Spectrum-X の adaptive routing や congestion control により、従来 Ethernet 比 2 倍の実効性能を達成。

InfiniBand が「専用高速道路（高速だが建設コストが高い）」なら、RoCE は「一般道を高速道路並みに改良したもの（コスト効率が良い）」。Spectrum-X は信号の最適化や車線拡張にあたる。

03データフロー：AI トレーニング 1 ステップの技術連携

分散 AI トレーニングの 1 イテレーションで、どの段階でどの技術が関与するかを追う。

AI トレーニングの基本的な流れ

AI モデルのトレーニングは「データを読む → 予測する → 正解と比較して誤差を計算する → 誤差を全 GPU で共有する → モデルを修正する」のサイクルを数百万回繰り返す作業。このサイクル 1 回分を「イテレーション」と呼ぶ。以下はそのイテレーションの中で、どのハードウェア・ソフトウェアがどの段階で登場するかを示す。

1. データロード

SSD などのストレージからトレーニングデータ（テキスト、画像など）を GPU メモリ (HBM) へ転送する。GPUDirect Storage により CPU メモリを経由せず直接ロードするため、CPU のボトルネックを回避できる。

GPUDirect Storage HBM メモリ CUDA

2. Forward / Backward Pass（計算）

GPU の Tensor Core でモデルの計算（行列演算）を実行する。Forward Pass でデータから予測を出し、Backward Pass で誤差（勾配）を計算する。モデルが大きすぎて 1 GPU に収まらない場合は Tensor Parallelism で複数 GPU に分割し、NVLink 経由で中間結果をやり取りする。

Tensor Core Transformer Engine NVLink NVSwitch CUDA

3. 勾配同期（ノード内）

各 GPU が計算した勾配（「モデルをどう修正すべきか」の情報）を、ラック内の 72 GPU 全てで合計・平均化する（AllReduce）。NVLink + NVSwitch の超高速接続（3.6 TB/s）を使うため、ナノ秒級の低遅延で完了する。NCCL が GPU 間の接続トポロジを自動検出し、最適な通信アルゴリズムを選択する。

NVLink + NVSwitch NCCL

4. 勾配同期（ノード間）

ラック間の勾配同期はネットワーク経由で行う。GPUDirect RDMA により GPU メモリから ConnectX SuperNIC へ直接データを送出し、CPU メモリのコピーを排除する。InfiniBand 環境では SHARP がスイッチ上で AllReduce の計算自体をオフロードし、ネットワーク上の転送データ量を約半分に削減する。Ethernet 環境では Spectrum-X の adaptive routing が輻輳を回避する。

ConnectX SuperNIC GPUDirect RDMA Spectrum-X / Quantum SHARP NCCL

5. パラメータ更新

全 GPU で同期された勾配を使い、各 GPU がオプティマイザ（Adam 等）でモデルのパラメータ（重み）を更新する。これでモデルが少し賢くなり、次のイテレーションへ進む。この 1〜5 の流れを数百万回繰り返すことで、モデルが完成する。

GPU (CUDA) HBM メモリ

04NCCL の通信パス選択ロジック

NCCL（NVIDIA Collective Communications Library）は AI トレーニングの集団通信において中心的な役割を担い、環境に応じて最適な通信パスを自動選択する。

NCCL とは

PyTorch などの AI フレームワークが「全 GPU の勾配を合計して」と命令すると、実際の通信処理を担当するのが NCCL。プログラマは通信の詳細を意識する必要がなく、NCCL が接続環境（NVLink があるか、InfiniBand か Ethernet か）を自動判定して最適な方法を選ぶ。いわば GPU 通信のカーナビのような存在。

07Ethernet vs InfiniBand：共通基盤と選択

AI ネットワークの 2 大プロトコル。競合しつつも ConnectX SuperNIC という共通ハードウェア基盤を持つ。

なぜ 2 つのプロトコルがあるのか

InfiniBand は HPC（スーパーコンピュータ）の世界で長年使われてきた高性能専用ネットワーク。最低遅延で SHARP による In-Network Computing も可能だが、専用機器が必要でコストが高い。Ethernet は世界中で使われている標準ネットワーク。従来は AI には不向きだったが、NVIDIA の Spectrum-X プラットフォームが RoCE adaptive routing 等の AI 最適化を追加し、InfiniBand に迫る性能を実現。Meta, Oracle などのクラウド企業が大規模に採用している。

重要なのは、NVIDIA の ConnectX SuperNIC が 両方に対応するデュアルプロトコル設計 であること。ユーザーは同じ NIC で InfiniBand にも Ethernet にも接続でき、環境に応じて選択できる。

11AI 推論パイプライン

トレーニングが「AI モデルを作る」フェーズなら、推論は「作ったモデルを使ってユーザーの質問に答える」フェーズである。2025年以降、推論が AI データセンターの主要なワークロードとなり、トレーニングとは根本的に異なる技術的課題と最適化戦略が求められている。

トレーニングと推論の根本的な違い

トレーニングはモデルの全パラメータの勾配を計算して更新する作業で、大量のメモリと計算能力が必要。推論は学習済みモデルに入力を与えて出力を得る作業で、勾配計算が不要な代わりに レイテンシ（応答速度）が最重要指標となる。

Prefill と Decode：推論の2つのフェーズ

LLM の推論は2つの全く異なる計算特性を持つフェーズに分かれる。この違いを理解することが推論最適化の出発点となる。

Prefill フェーズ（プロンプト処理）

ボトルネック：計算能力（Compute-bound）
処理：入力トークン全体を並列に行列乗算
特徴：高い Arithmetic Intensity（演算密度）
指標：TTFT（最初のトークンが出るまでの時間）
GPU 利用率：Tensor Core 高稼働
計算バウンド大バッチ処理向き

Decode フェーズ（トークン生成）

ボトルネック：メモリ帯域幅（Memory-bound）
処理：1トークンずつ逐次生成
特徴：低い Arithmetic Intensity
指標：ITL（トークン間の生成時間）
GPU 利用率：メモリ帯域が律速
メモリバウンド低レイテンシ必須
例え話：Prefill と Decode の違い

Prefill は「試験問題用紙をまとめて読み込む」作業 — 大量のテキストを一度に処理するので、読む速度（計算能力）が勝負。Decode は「回答を1文字ずつ書く」作業 — 毎回ノート（KV キャッシュ）を見返す必要があるので、ノートをめくる速度（メモリ帯域幅）が勝負。

KV キャッシュ：推論のメモリボトルネック

KV キャッシュとは何か

Transformer の Attention 機構では、各トークンの出力を計算するために過去の全トークンの Key (K) と Value (V) を参照する。毎回再計算するのは非効率なので、一度計算した K, V をキャッシュして再利用する。これが KV キャッシュ。キャッシュがないと N トークンの生成に O(N²) の計算が必要だが、キャッシュにより O(N) に削減できる。

KV キャッシュはコンテキスト長とバッチサイズに比例して爆発的に増大する。以下は Llama 2 70B (FP16) の例。

品質を一切低下させない高速化手法

HBM4 の技術的進化

HBM4 はバス幅を 1024-bit から 2048-bit に倍増し、独立チャネル数を 16 から 32 に倍増。スタックあたり帯域幅は 2+ TB/s に達し、Rubin GPU では 8 スタックで 22 TB/s のメモリ帯域幅を実現する。SK Hynix が世界初の HBM4 開発を完了し、2026 Q3 量産開始予定。

なぜメモリ帯域幅が AI 推論のボトルネックか

LLM 推論の Decode フェーズでは、1 トークン生成のためにモデル全パラメータ（70B = ~140 GB）を HBM から読み込むが、実行される演算量は 140 GFLOP に過ぎない。Arithmetic Intensity は 1 FLOP/Byte で、B200 の転換点 562.5 FLOP/Byte と比較して圧倒的に低い。つまり LLM の Decode は完全に Memory-bound であり、メモリ帯域幅の向上こそが推論性能を直接改善する。

GPUDirect テクノロジー

メモリ階層間のデータ転送を最適化し、CPU のバウンスバッファを排除する技術群。GPU メモリ（HBM）を中心としたフラットなデータアクセスを実現する。

GPUDirect P2P

GPU 間の HBM を NVLink で直接転送。CPU メモリを経由せず、マルチ GPU トレーニング/推論で GPU 間 HBM を仮想的に統合されたメモリプールとして活用可能。

MoE（Mixture of Experts）は GPT-4、DeepSeek-V3/R1、Gemini など主要フロンティアモデルが採用するアーキテクチャであり、AllReduce とは異なる All-to-All 通信パターンを必要とする。数万 GPU のクラスタでは数時間に1回の障害が不可避であり、高可用性の確保が大きな課題。本セクションでは技術と運用の両面から実態を整理する。

MoE（Mixture of Experts）の仕組み

Dense モデル vs MoE モデル

Dense モデルは全パラメータを全入力に対して使用する。MoE モデルは入力ごとに一部の「Expert」（専門家サブネットワーク）のみを活性化する条件付き計算を導入。これにより、モデル容量（知識量）を増やしながら計算コストを一定に保てる。例えば DeepSeek-V3 は総パラメータ 671B だが活性化パラメータは 37B — 全体の 5.5% しか使わない。

例え話：MoE = 総合病院

Dense モデルが「全科目を1人で診る万能医師」なら、MoE は「専門医が何人もいる総合病院」。患者（入力トークン）が来ると、受付の看護師（Router）が症状に応じて適切な専門医（Expert）を割り振る。全医師を同時に呼ぶ必要がないので、病院全体の効率が大幅に上がる。

All-to-All 通信：MoE 固有のネットワーク課題

なぜ AllReduce ではなく All-to-All が必要か

Dense モデルの Data Parallelism では全 GPU が同じモデルのコピーを持ち、勾配を AllReduce で集約する。通信パターンは予測可能。一方、MoE の Expert Parallelism では Expert が異なる GPU に分散配置され、各トークンは Router によって動的に異なる Expert（= 異なる GPU）に送られるため、全 GPU が全 GPU にデータを送る All-to-All 通信が必要になる。通信量は不均一かつ予測困難で、ネットワーク輻輳のリスクが高い。

例え話：AllReduce vs All-to-All

AllReduce は「全員が同じ教科書のテストを受けて、平均点を計算して全員に配る」— 全員が同じ種類のデータを交換するので整然としている。All-to-All は「30 人のクラスで全員が全員に異なる手紙を出す」— 誰から誰に何を送るかは毎回変わり、郵便局（ネットワーク）が混雑しやすい。

NVL72 での MoE 最適化

Wide Expert Parallelism

TensorRT-LLM の Wide-EP は NVL72 のラックスケール NVLink ドメイン（130 TB/s）を最大限に活用。EP=32 で EP=8 比 最大 1.8x のスループット向上。重みロード圧力を削減し、GroupGEMM 効率を改善。

Hybrid-EP ライブラリ

NVLink / InfiniBand / Ethernet / RDMA を統合活用した Expert Parallel 通信最適化ライブラリ。わずか 4 SM で NVLink 帯域幅をほぼ最大化。DeepSeek-V3 で 約 14% の性能改善。

ConnectX-8 All-to-All 最適化

PCIe Gen6 + 800 Gb/s ネットワーキングを単一デバイスに統合。GPU あたり最大 400 Gb/s、NCCL All-to-All 性能で前世代比 2x 高速化。

MoE 推論性能

Blackwell NVL72 vs Hopper H200: MoE モデルで 10x の性能向上、トークンあたりコスト 1/10 に削減。DeepSeek-R1: 単一 DGX B200 で 250 TPS/user、最大 30,000 TPS。

チェックポイント技術

同期チェックポイント

全 GPU がトレーニングを停止し、モデルパラメータ・オプティマイザ状態をストレージに保存。大規模モデルのチェックポイントサイズは数百 GB〜数 TB。全 GPU がアイドル状態になるためオーバーヘッドが大きい。

非同期チェックポイント（現在の主流）

GPU メモリから CPU メモリにコピー（短時間のブロッキング）した後、CPU スレッドがバックグラウンドでストレージに書き出し。GPU はすぐにトレーニングを再開。GPU ブロッキング時間を同期比 10x 以上削減。Meta Llama 3 では 4 分ごと・2.5 秒で実行。

NVIDIA Mission Control

大規模 AI クラスタの運用管理プラットフォーム。ハードウェア障害の自動検出・修復と、AI トレーニングジョブの自動回復を統合的に提供する。

AHR（自動 HW 回復）

ハードウェア障害の自動検出と修復。NVLink リンク状態の継続監視、障害リンクの切り離し、RMA 後の自動復帰。非同期操作でラック間の並列修復を実行。

AJR（自動ジョブ回復）

AI トレーニングジョブの自動障害検出・隔離・回復。FACT エンジンがシステムログ・ネットワークメトリクスを分析して障害原因を自動特定。手動介入を大幅に削減。

NetQ テレメトリ

NVLink と Ethernet の統合可視化。リアルタイムのネットワークファブリック監視。パケットロス・HW 障害・設定エラーの即時検出。Spectrum-X からのテレメトリデータも統合。
