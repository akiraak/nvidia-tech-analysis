01AI 半導体エコシステムの全体像

AI の急成長に伴い、NVIDIA GPU 以外にも多数の AI 半導体が登場している。各社のアプローチは大きく異なり、Google のように独自チップ + 独自インフラで完全に垂直統合する企業もあれば、AMD のように標準規格で NVIDIA の代替を目指す企業もある。ここでは8つの主要プレイヤーの全体像を概観する。

ASIC とは

ASIC（Application-Specific Integrated Circuit）は、特定の用途に特化して設計されたチップのこと。汎用的に何でもできる GPU と異なり、AI の推論やトレーニングなど特定の処理だけを極めて高効率に実行する。スマートフォンで例えると、GPU は「何でもできるスマホ」、ASIC は「電話だけに特化した固定電話」のようなもの。専用設計であるぶん、特定用途では圧倒的に効率が良い。

このページを読むための基礎知識

AI データセンターの技術を理解するために押さえておくべき4つの概念を解説する。これらの用語は以降のセクションで頻繁に登場する。

Scale-up vs Scale-out

チップ間接続の2つの階層

Scale-up は1台のサーバ内でチップ同士を高速に接続すること。Scale-out は複数のサーバをネットワークでつないで規模を拡大すること。Scale-up は高速だが規模に限界があり、Scale-out はスケールするがネットワーク遅延がボトルネックになる。

マンションに例えると、Scale-up は「同じフロア内の部屋の移動」（廊下を歩くだけ）、Scale-out は「別の建物への移動」（外に出て道を歩く必要がある）。

ASIC vs GPU

専用設計 vs 汎用チップ

GPU はグラフィックスから AI まで幅広い用途に対応する汎用チップ。ASIC は AI 推論やトレーニングなど特定の処理に特化した専用チップ。ASIC は特定用途で GPU を上回る効率を持つが、用途が限定される。

GPU は「万能包丁」で何でも切れるが、ASIC は「刺身包丁」のように特定の用途で圧倒的に切れ味が良い。

プロプライエタリ vs オープン標準

ロックイン vs マルチベンダ

プロプライエタリ（例: NVLink, ICI）は1社が規格を制御し、最高の性能を発揮するが他社製品との互換性がない。オープン標準（例: Ethernet, UALink）は複数ベンダが実装可能で選択の自由があるが、性能は専用設計に劣ることが多い。

プロプライエタリは「専用軌道の新幹線」（速いが、JR の線路しか走れない）。オープン標準は「標準軌の在来線」（どの会社の列車も走れるが、速度は新幹線に及ばない）。

垂直統合 vs 水平分業

Google/AWS 型 vs AMD/Intel 型

垂直統合（Google, AWS）はチップからソフトウェアまで自社で一貫して設計し、最適化を最大化する。水平分業（AMD, Intel）は標準規格のコンポーネントを組み合わせ、ユーザーに選択の自由を提供する。

垂直統合は「Apple の iPhone」（ハード・OS・サービスをすべて自社設計で、体験は統一されるが選択肢がない）。水平分業は「Android スマホ」（様々なメーカーの組み合わせが選べるが、最適化は Apple ほどではない）。
各社の概要を見たところで、次はそれぞれがどのようにデータセンターを構成しているか — チップ間の接続方式、サーバ構成、スケーリング階層を具体的に見ていこう。

02主要プラットフォーム別データセンター構成

AI データセンターでは「チップ単体の性能」よりも「数千〜数万のチップをどう接続するか」が性能を決定する。同じチップでも接続方式が異なれば、全体の性能は大きく変わる。ここでは Google TPU、AMD Instinct、AWS Trainium、Intel Gaudi の4大プラットフォームについて、データセンター構成の詳細を比較する。

なぜ接続方式が重要なのか

AI のトレーニングでは、数千個のチップが同時に計算を行い、その結果を全チップで共有する必要がある（これを AllReduce と呼ぶ）。この「結果の共有」がボトルネックになるため、チップ間の接続速度と方式がデータセンター全体の性能を左右する。接続が遅いと、いくらチップ単体が高性能でも、待ち時間でトレーニング全体が遅くなる。

Google TPU — 完全垂直統合型

なぜ Google は独自チップを作るのか？ Google は世界最大の AI モデル（Gemini）を自社で開発・運用しており、NVIDIA GPU に依存すると調達コスト・供給量の両面でリスクがある。独自チップを設計することで、自社の AI ワークロードに最適化されたインフラをコントロールでき、GCP の差別化にもつながる。

Google 独自のネットワークインフラで複数サイトを接続。全レイヤーが Google 独自技術で構成されており、外部ベンダへの依存がない。

GCP 専用
たとえ話で理解する ICI と OCS

ICI（3D Torus）は、格子状に並んだ教室を廊下で直接つなぐ学校のようなもの。各教室は上下左右の教室と直接行き来でき、事務室（スイッチ）を経由する必要がない。

OCS（光回路スイッチ）は、回転する鏡で光の道を切り替える信号機。配線を物理的に変えることなく、「今はこの教室とあの教室をつなぐ」「次はこの組み合わせ」と動的に接続先を変更できる。

ソフトウェア: JAX + Pathways

  • JAX: Google 製の関数型 ML フレームワーク。PyTorch の代替として TPU に最適化
  • XLA: コンパイラが Python コードを自動で TPU 向けに最適化
  • GSPMD: 手動で並列化を設定しなくても、コンパイラが自動で複数 TPU に処理を分散
  • Pathways: ジョブスケジューリングの分散ランタイム

制約と注意点

  • GCP でのみ利用可能 — オンプレミスでは使えない
  • PyTorch のネイティブサポートは限定的（JAX が前提）
  • Google のエコシステムへの完全なロックイン
  • 一方で、電力効率は全チップ中最高（29.4 GFLOPS/W）

AMD Instinct — オープンエコシステム型

なぜ AMD はオープン路線なのか？ NVIDIA のように独自のインターコネクトとソフトウェアでエコシステムを囲い込む戦略は、後発の AMD には難しい。代わりに AMD は標準 Ethernet と ROCm（オープンソースの GPU プログラミング基盤）で「選択の自由」を提供し、NVIDIA へのロックインを嫌う顧客を取り込む戦略を採る。

Scale-up: xGMI Full Mesh

1台のサーバ内で 8基の GPU を xGMI（Infinity Fabric）でフルメッシュ接続。各 GPU 間は 128 GB/s のリンクで直結し、集約帯域は 896 GB/s。UBB 2.0（Universal Baseboard）に OAM 規格で搭載。

  • 8 GPU フルメッシュ（全GPU が直接通信可能）
  • 128 GB/s/リンク × 7本 = 896 GB/s 集約
  • NVIDIA NVLink に相当するが、帯域幅は劣る

Scale-out: Rail-Optimized Ethernet

各 GPU に 専用の NIC（400G）を 1:1 で配置し、GPU ごとに独立したネットワークレーン（Rail）で通信する構成。GPU が NIC を共有する従来方式と比べてボトルネックが解消される。

  • GPU 0 → NIC 0 → Leaf Switch Rail 0
  • GPU 1 → NIC 1 → Leaf Switch Rail 1 ...
  • 検証済み NIC: ConnectX-7、Broadcom Thor-2、AMD Pollara 400G
  • リファレンス設計: 128〜8,192 GPU クラスタ
たとえ話で理解する Rail-Optimized

Rail-Optimized は、8車線の高速道路で各車線にそれぞれ専用の料金所がある構成。従来方式は全車線の車が1つの料金所に合流するため渋滞が発生するが、Rail-Optimized では各車線が独立して流れるため渋滞が分散される。

ソフトウェア: ROCm + RCCL

  • ROCm 7.x: PyTorch 2.9、JAX 0.8.0 対応のオープンソース GPU SDK
  • HIP: CUDA コードを AMD 向けに変換するツール（hipify-perl で自動変換）
  • RCCL: NCCL 互換の集団通信ライブラリ（API は同じだが内部最適化が異なる）
  • 推論は vLLM on ROCm が推奨構成

課題と制約

  • RCCL の最適化が NCCL に追いついておらず、AllReduce 性能はNVIDIA の約50%
  • ROCm のツール群（プロファイラ、デバッガ）は CUDA に比べ未成熟
  • FlashAttention の ROCm 対応は遅延することが多い
  • ただし MI300X の 192GB メモリは H100 (80GB) の 2.4倍で、大規模モデルの推論に強い

なぜ AWS は自社チップを開発するのか？ AWS は世界最大のクラウドプロバイダとして、NVIDIA GPU の調達コストと供給不足のリスクを最も強く感じている。自社チップを開発することで、NVIDIA への依存度を下げ、顧客により低コストの AI インフラを提供できる。Project Rainier（投資額110億ドル）は、この戦略の集大成。

Intel Gaudi — 標準 Ethernet 統合型

なぜ Gaudi は NIC を内蔵するのか？ 通常の AI アクセラレータは外部に NIC（ネットワーク接続用チップ）を必要とし、その分コストと電力が増加する。Gaudi は NIC をチップ内に24ポート内蔵することで、外部 NIC のコストを削減し、標準 Ethernet でロックインなしに Scale-up を実現する。

Scale-up: オンチップ直接接続

8基の Gaudi 3 OAM が基板上で スイッチ不要の All-to-All 直接接続。各カードから他の7枚へ 3 × 200GbE = 600 Gbps のリンクを持つ。24ポートのうち 21ポートを Scale-up に使用し、67.2 Tbps の双方向帯域を実現。

Scale-out: 標準 Ethernet

残り 3ポートを Scale-out に使用し、外部 6 × 800GbE OSFP で Leaf-Spine Fat-tree トポロジに接続。3:1 のオーバーサブスクリプション比で、最大 1,024 アクセラレータまで拡張可能。

RoCE（ロッキー）とは

RoCE（RDMA over Converged Ethernet）は、Ethernet ネットワーク上で CPU を介さずにメモリ間で直接データを転送する技術。通常のネットワーク通信では CPU がデータの送受信を管理するが、RoCE では NIC が直接メモリにアクセスするため、CPU のオーバーヘッドがなくなり、レイテンシが大幅に低下する。Gaudi はこの RoCE をチップ内に内蔵している。

Gaudi の課題

Falcon Shores（Gaudi 後継の高性能チップ）は 2025年1月にキャンセルされた。売上目標5億ドル未達で事業計画を見直し。次世代 Jaguar Shores（2026年以降）でラックスケールシステムへの方向転換を図るが、Intel の AI アクセラレータ事業の将来は不透明。

大手4社のデータセンター構成を見てきたが、AI 半導体市場にはまったく異なるアプローチで参入している新興企業もある。次のセクションでは、GPU という概念そのものを再定義しようとしている企業群を紹介する。

03新興 AI アクセラレータ

前のセクションで紹介した4社は、いずれも従来の「チップ + メモリ + ネットワーク」という GPU に近い設計を採用している。しかし、AI の急速な進化は「GPU の形にこだわる必要はないのでは？」という問いを生んだ。ここでは、ウェーハ全体を1チップにする Cerebras、待ち時間ゼロの推論を実現する Groq、3層メモリで超大規模モデルに対応する SambaNova、RISC-V で AI チップの民主化を目指す Tenstorrent を紹介する。

たとえ話で理解する新興アクセラレータの設計思想

Cerebras（Weight Streaming）は、百科事典を1巻ずつ棚から取り出して読む方式。従来の GPU は「全巻を机に広げる」（HBM にすべて載せる）が、Cerebras は巨大な机（44GB SRAM）に必要な1巻だけを載せ、読み終わったら次の巻に差し替える。

Groq（決定論的実行）は、工場の組立ライン。各ステップの所要時間がコンパイル時に完全に確定しており、「前の工程が終わるのを待つ」という無駄がゼロ。通常の GPU は「混雑するレストラン」で、注文の順番も調理時間も予測不能。

SambaNova（3層メモリ）は、3段の引き出し。一番上（SRAM）は手が届きやすいが小さい。真ん中（HBM）はそこそこのサイズ。一番下（DDR）は巨大だが取り出すのに時間がかかる。使用頻度に応じてデータを自動で最適な引き出しに配置する。

ここまでで各社の独自技術を見てきた。では、これらの非 NVIDIA チップは NVIDIA のネットワーク技術（NIC、スイッチ、インターコネクト）と組み合わせることができるのか？ この互換性こそが、実際のデータセンター構築における最大の検討事項の一つだ。

04NVIDIA 技術との互換性マトリクス

AI データセンターの構築では「どのチップを使うか」だけでなく、「どのネットワーク機器と組み合わせるか」が重要な検討事項となる。NVIDIA は GPU だけでなく、NIC（ConnectX）、スイッチ（Spectrum-X, Quantum）、インターコネクト（NVLink）、通信ライブラリ（NCCL）など、データセンターインフラ全体を提供している。ここでは、NVIDIA の GPU 以外の技術を他社 AI 半導体と組み合わせた場合に何が可能で何が不可能かを整理する。

なぜ互換性が問題になるのか

NVIDIA のネットワーク技術は標準規格（PCIe、Ethernet、InfiniBand）に準拠しているため、物理的には他社チップと接続できる場合が多い。しかし、性能を最大限引き出すにはソフトウェア（NCCL、GPUDirect）との垂直統合が必要であり、これが NVIDIA GPU に限定されている。つまり「つながる」と「最高性能で動く」は別問題。

たとえ話で理解する NVLink Fusion

NVLink Fusion は、他社の電車を NVIDIA の線路（NVLink）に乗り入れさせる仕組み。ただし条件があり、NVIDIA の駅（CPU または GPU）を経由する必要がある。AWS Trainium4 が最初の大規模な「相互乗り入れ」事例で、NVIDIA の高速線路を使いつつ自社の車両（Trainium チップ）を走らせる。

AllReduce 性能比較（実測データ）

マルチノード AllReduce、メッセージサイズ 16MiB〜256MiB での相対性能（Semi Analysis 実測）。

AllReduce とは

分散 AI トレーニングでは、各チップが計算した「勾配」（モデルの修正量）を全チップで集約・平均し、結果を全チップに配布する必要がある。この操作を AllReduce と呼ぶ。AllReduce が遅いとチップは計算を終えても結果の共有待ちで止まるため、トレーニング全体のスピードが落ちる。以下のグラフは、各構成でこの AllReduce がどれだけ速いかを比較したもの。

性能ギャップの原因

MI300X + RoCEv2 が InfiniBand Non-SHARP の約半分にとどまる主因は RCCL のネットワーク最適化不足。興味深いことに、NIC ハードウェアが同じ ConnectX-7 でも、ソフトウェアスタック（NCCL vs RCCL）の差が性能を決定する。つまり、チップの性能だけでなく通信ソフトウェアの品質が決定的に重要。Spectrum-X が IB に近い性能を達成できるのは NCCL との垂直統合による最適化の恩恵。

NVLink 6 の強み

  • 帯域幅: 3,600 GB/s（圧倒的な速度）
  • NVIDIA の NVSwitch で 72 GPU を non-blocking 接続
  • NVLink Fusion でサードパーティに限定的開放

UALink 1.0 の意義

帯域幅の差は圧倒的だが、オープン性に価値がある

NVLink 6 は UALink 1.0 の 18倍の帯域幅を持つ。数字だけ見れば NVLink の圧勝だが、これは NVIDIA エコシステムへの完全なロックインを意味する。UALink はオープン標準として複数ベンダが実装でき、2.0 以降の帯域幅向上が期待される。歴史的にも、Ethernet が当初は専用プロトコル（Token Ring、FDDI）に性能で劣りながらも、オープン性により市場を制覇した前例がある。UALink が同じ道を辿るかは、2026〜2028年の市場動向で明らかになる。

ハードウェアの接続方式を比較してきたが、実際のプラットフォーム選定では ソフトウェアの成熟度 がより重要な決定要因となることが多い。CUDA エコシステムからの移行コストが、技術選定の最大の障壁だ。

06ソフトウェアエコシステム比較

AI チップのハードウェア性能が拮抗しつつある中、勝敗を分けるのはソフトウェアエコシステムの充実度だ。NVIDIA の CUDA は15年以上の蓄積があり、PyTorch、TensorFlow、主要な AI フレームワークのほぼすべてが CUDA を前提に開発されている。各社がこの「CUDA の壁」をどう乗り越えようとしているかを比較する。

CUDA の壁とは

世界中の AI 研究者・エンジニアの大半が CUDA でコードを書いている。NVIDIA 以外のチップに移行するには、既存コードの書き換えが必要になる場合が多く、これが巨大な移行コストとなる。ソフトウェア資産の蓄積は、ハードウェアの性能差よりも強力なロックイン要因。

Optimum-Habana SynapseAI

集団通信ライブラリの群雄割拠

分散 AI トレーニングでは AllReduce 等の集団通信が性能のボトルネックになる。NVIDIA の NCCL が事実上の標準だが、各社が独自のライブラリを開発しており、新興の代替手段も登場している。

NCCL (NVIDIA)

業界標準の集団通信ライブラリ。CUDA 専用。SHARP（InfiniBand スイッチ上で集約処理を実行する技術）+ NVLink と垂直統合されており、NVIDIA 環境では圧倒的な性能。

RCCL (AMD)

NCCL のフォーク（派生版）で API 互換。ただし SHARP のようなネットワーク側の最適化がなく、NCCL 比で性能が劣る。AMD が重点的に改善中。

MSCCL++ (Microsoft)

Microsoft が Azure 向けに開発。NCCL 比 最大 2.8倍、RCCL 比 最大 3.8倍の高速化を達成。Azure の大規模クラスタで実績。

UCCL (UC Berkeley)

NCCL/RCCL のドロップイン代替（コード変更なしで差し替え可能）。マルチベンダ NIC に対応し、ConnectX、Broadcom、EFA のいずれでも動作。

HetCCL (研究)

学術研究発の革新的ライブラリ。AMD + NVIDIA の GPU を混在させたトレーニングに対応し、同種 GPU のみの構成と比べて 98% のスループットを達成。

OneCCL (Intel)

Intel oneAPI エコシステムの一部。Gaudi および Xeon 向けに最適化された集団通信ライブラリ。標準的な MPI インターフェースを提供。

ソフトウェアの成熟度に加えて、データセンター構築の意思決定で最も重要なのがコストと電力効率だ。特に 2025年以降、電力供給がデータセンター拡張のボトルネックとなっており、GFLOPS/W（ワットあたりの演算性能）の比較は戦略的意味を持つ。

なぜ Google Ironwood の電力効率が突出しているのか

08今後の展望

AI データセンター市場は急速に変化しており、2026〜2028年にかけて大きな構造変化が予想される。NVIDIA の圧倒的な優位は続くのか、それとも対抗勢力が勢力図を塗り替えるのか。3つの大きなトレンドから今後の方向性を読み解く。

NVLink Fusion の戦略的意義

NVIDIA がインターコネクトをサードパーティに開放することで、カスタム ASIC 市場を NVIDIA エコシステムに取り込む戦略。AWS Trainium4 が最初の大規模事例。

これは一見 NVIDIA の譲歩に見えるが、実質的には「競合チップですら NVIDIA のインフラに乗る」という構図を作り、NVLink のデファクトスタンダード化を狙うもの。NVIDIA は GPU を売らなくても、インフラ層で収益を得られる。

UALink とオープン標準の挑戦

AMD/Intel/Google/Meta が推進するオープン標準。現時点では NVLink の 1/18 の帯域幅だが、歴史的に見ればオープン標準が最終的に市場を制覇した例は多い（Ethernet vs Token Ring、USB vs 各社独自端子）。

鍵は UALink 2.0 以降の帯域幅向上スピード。NVLink との差が縮まれば、マルチベンダ対応のメリットが大きくなり、大規模クラウドプロバイダの採用が加速する可能性がある。

推論 ASIC の台頭

Cerebras + OpenAI（100億ドル+の推論契約）、AWS Trainium の急速な成長、Groq の NVIDIA 統合。推論市場では汎用 GPU から専用 ASIC へのシフトが加速している。

推論はトレーニングと異なり、モデルが確定した後の「繰り返し処理」であるため、汎用性よりもコスト効率・電力効率・レイテンシが重視される。専用設計の ASIC がこれらの指標で GPU を上回るのは当然であり、この流れは不可逆的。

NVIDIA のエコシステムロックインの構造

NVIDIA のデータセンター技術は、ハードウェアレベルでは多くが標準規格（PCIe、Ethernet、InfiniBand）に準拠しており他社製品と接続可能。しかし、ソフトウェアレイヤ（NCCL、CUDA、GPUDirect）での最適化が NVIDIA GPU に限定されていることで、実質的なエコシステムロックインを形成している。

つまり「物理的につながる」と「最高性能で動く」は別の問題であり、性能の最大化には NVIDIA GPU が必要という構造が NVIDIA の競争優位の核心。これを崩すには、ソフトウェアスタックの代替（ROCm、JAX/XLA、Neuron SDK）の成熟と、オープンな集団通信ライブラリ（UCCL、HetCCL）の普及が不可欠。

トレーニング: NVIDIA GPU が当面支配的。CUDA + NVLink + NCCL の垂直統合を崩すのは困難で、2028年時点でも 90%+ のシェアが予想される。

推論: ASIC の急速な台頭。コスト効率と電力効率が重視される推論ワークロードでは、TPU・Trainium・カスタムチップが 70-75% のシェアを獲得する可能性。
